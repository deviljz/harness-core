# Regression 004: 上传进度卡在 100% + 「+」按钮转圈被误读

## 背景

2026-06 选图打卡乐观渲染：缩略图叠加上传百分比。
百分比只反映"字节灌进 socket"(HTTP sent)，到 100% 后还要等服务器落盘+建记录+列表刷新
才真正结束。这段空窗里缩略图一直显示「100%」，且「+」按钮在 busy 期间渲染成转圈 loading，
视觉上排在缩略图右边被用户误读成"第二张图在转"。
开发者用 323KB 小图 + 局域网秒传验证，空窗被压成 ~0ms，问题完全没暴露。
(commit a644a80 / 4ede293 修复：加 processing 态显示「处理中…」，+ 按钮 busy 不转圈)

## 预期行为

review 应发现：进度到 100% 不等于完成（异步中间态缺失），且测试用小图/秒传无法覆盖
"100%→服务器响应"的空窗——应注入慢依赖(延迟 resolve 的 fake service)断言中间态。
判定 consistent=false。

## 为何重要

异步/上传/乐观渲染类的"慢依赖中间态"盲区：用秒传验证等于把唯一能暴露 bug 的条件删掉。
必须注入"慢依赖"把中间态变成可断言的测试，而不是依赖真实网络恰好够快。
